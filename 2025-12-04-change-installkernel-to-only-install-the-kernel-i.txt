     Plan: Process-Safe Idempotent Kernel Installation

     Current Situation

     The DataFerret kernel is unconditionally reinstalled every time:
     - Jupyter server starts with the extension enabled
     - Any CLI command is run (data_ferret, data_ferret_optimize, etc.)

     Location: data_ferret/util/kernel_installer.py

     Current behavior:
     - install_kernel_spec() function always calls install_kernel()
     - No check for existing installation
     - Uses KernelSpecManager.install_kernel_spec() with replace=True
     - Previous check code exists but is commented out (lines 90-95)

     Problems:
     1. Overhead: Every JupyterLab startup and CLI command reinstalls 
     (100ms+ each time)
     2. Race conditions: Multiple processes can conflict during installation
     3. Development friction: Constant reinstallation during development 
     iterations

     Chosen Approach: Smart Check + Conditional Install

     Why this approach:
     - ✅ Fast path: ~1ms check when already installed
     - ✅ Auto-updates when Python path changes (virtualenv switches)
     - ✅ Perfect for development workflow with pip install -e .
     - ✅ Process-safe with file locking
     - ✅ Minimal runtime overhead

     Source Change Detection Strategy

     Primary check: Python executable path
     - Compare sys.executable with installed kernel's argv[0]
     - Handles virtualenv switches automatically (most common case)
     - Resolved paths (handles symlinks)

     Future enhancement (not implemented now):
     - Modification time checking for kernel source files
     - Keeps initial implementation simple

     Acceptable trade-off: Source file changes without Python path changes 
     require manual reinstall. This is fine for development - when you 
     modify kernel source, you're likely testing and can restart manually.

     Design Goals

     1. Idempotent: Only install if kernel is not already correctly 
     installed
     2. Process-safe: Multiple processes can safely attempt installation 
     concurrently
     3. Correct verification: Check that installed kernel matches current 
     spec (Python path)
     4. Minimal overhead: Fast check for already-installed case
     5. Robust: Handle edge cases (partial installs, stale locks)

     Implementation Approach

     1. Two-Level Check Strategy

     Fast path (no lock needed):
     - Check if kernel exists using KernelSpecManager.get_kernel_spec()
     - Verify kernel.json has correct Python executable path
     - If both pass, skip installation entirely

     Slow path (lock-protected):
     - Acquire file-based lock
     - Double-check kernel status (another process may have just installed 
     it)
     - Install kernel if still needed
     - Release lock

     2. Lock Mechanism

     Use file-based locking with the filelock library:
     - Platform-independent (Windows, Linux, macOS)
     - Handles stale locks automatically with timeout
     - Small, well-tested dependency

     Lock location: Inside Jupyter's data directory to avoid permissions 
     issues
     - Use jupyter_core.paths.jupyter_data_dir() to get base path
     - Create lock file at: {jupyter_data_dir}/locks/ferret_kernel.lock

     Timeout: 30 seconds (generous for slow filesystems)

     3. Verification Logic

     Check if kernel needs installation/update by:
     1. Call KernelSpecManager().get_kernel_spec("ferret_kernel")
       - If raises exception: needs installation
     2. Read installed kernel.json
     3. Check if first argv element matches sys.executable
       - If different: Python environment changed, needs reinstallation
     4. Optional: Check if source files are newer than installed files

     Files to Modify

     Primary File: data_ferret/util/kernel_installer.py

     Changes:
     1. Add filelock import
     2. Create new function is_kernel_installed_correctly(kernel_name: str) 
     -> bool:
       - Check existence via get_kernel_spec()
       - Verify Python path matches sys.executable
       - Return True if already correct, False otherwise
     3. Modify install_kernel_spec(kernel_name: str, spec_source_dir: Path):
       - Fast check: if is_kernel_installed_correctly(), return early
       - Acquire lock at {jupyter_data_dir}/locks/ferret_kernel.lock
       - Double-check again inside lock
       - Call install_kernel() if needed
       - Release lock (via context manager)
     4. Keep install_kernel() unchanged (it's already correct)

     Dependencies: pyproject.toml

     Function: is_kernel_installed_correctly(kernel_name: str) -> bool

     def is_kernel_installed_correctly(kernel_name: str) -> bool:
         """Check if kernel is installed and has correct Python path."""
         try:
             ksm = KernelSpecManager()
             spec = ksm.get_kernel_spec(kernel_name)

             # Read the installed kernel.json
             kernel_json_path = Path(spec.resource_dir) / "kernel.json"
             if not kernel_json_path.exists():
                 return False

             with open(kernel_json_path, 'r') as f:
                 kernel_data = json.load(f)

             # Check if Python executable matches current environment
             if not kernel_data.get('argv'):
                 return False

             installed_python = kernel_data['argv'][0]
             current_python = sys.executable

             # Compare resolved paths (handle symlinks)
             return Path(installed_python).resolve() == 
     Path(current_python).resolve()

         except (NoSuchKernel, FileNotFoundError, json.JSONDecodeError, 
     KeyError):
             return False

     Function: install_kernel_spec(kernel_name: str, spec_source_dir: Path)

     def install_kernel_spec(kernel_name: str, spec_source_dir: Path):
         """Install kernel spec with process-safe idempotent behavior."""
         from filelock import FileLock, Timeout
         from jupyter_core.paths import jupyter_data_dir

         # Fast path: check if already correctly installed
         if is_kernel_installed_correctly(kernel_name):
             log(f"Kernel '{kernel_name}' already correctly installed")
             return

         # Slow path: acquire lock and install
         lock_dir = Path(jupyter_data_dir()) / "locks"
         lock_dir.mkdir(parents=True, exist_ok=True)
         lock_file = lock_dir / f"{kernel_name}.lock"

         lock = FileLock(lock_file, timeout=30)

         try:
             with lock:
                 # Double-check: another process may have just installed it
                 if is_kernel_installed_correctly(kernel_name):
                     log(f"Kernel '{kernel_name}' was installed by another 
     process")
                     return

                 log(f"Installing kernel spec '{kernel_name}'")
                 install_kernel(kernel_name, spec_source_dir)
                 log(f"Kernel '{kernel_name}' installation complete")

         except Timeout:
             log(f"Warning: Timeout acquiring lock for kernel installation. 
     "
                 f"Another process may be installing. Continuing anyway...")
             # Don't fail - the other process will likely complete 
     installation

     Edge Cases Handled

     1. Race condition: Double-check pattern prevents duplicate installs
     2. Stale lock: 30-second timeout prevents indefinite blocking
     3. Python environment change: Verification checks Python path
     4. Partial installation: If installation fails, next attempt will retry
     5. Lock directory doesn't exist: Created with mkdir(parents=True, 
     exist_ok=True)
     6. Symlinked Python: Use Path.resolve() to compare actual paths
     7. Lock timeout: Log warning but don't fail (trust other process)

     Testing Approach

     Manual testing scenarios:
     1. Clean install: Delete kernel, run command, verify installation
     2. Already installed: Run twice, verify second run skips installation
     3. Python path change: Install with one Python, switch venv, verify 
     reinstall
     4. Concurrent processes: Start multiple data_ferret commands 
     simultaneously, verify no conflicts
     5. Lock timeout: Artificially hold lock, start another process, verify 
     graceful handling

     Can verify with:

     Benefits

     1. Performance: Fast path check takes ~1ms vs ~100ms for full install
     2. Safety: File locks prevent concurrent modification
     3. Correctness: Verifies Python path matches current environment
     4. Simplicity: Minimal code changes, leverages existing Jupyter APIs
     5. Robustness: Handles edge cases gracefully

     Risks and Mitigations

     | Risk                   | Mitigation                              |
     |------------------------|-----------------------------------------|
     | Lock file permissions  | Use Jupyter's data dir (user-writable)  |
     | Lock timeout too short | 30s is generous; log warning on timeout |
     | Different filesystems  | filelock library handles cross-platform |
     | Disk full              | Same failure mode as current code       |

     Alternative Approaches Considered

     1. Jupyter's replace=True: Current approach, but reinstalls every time
     2. PID-based locking: Less robust, doesn't handle crashes
     3. Database lock: Overkill for this use case
     4. No verification: Faster but might miss environment changes

     Chosen approach balances simplicity, safety, and correctness.
