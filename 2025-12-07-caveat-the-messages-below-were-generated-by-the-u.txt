     Overview

     Create a new server command TestDependencies that executes all cells in
      a notebook while comparing static vs. dynamic dependency tracking,
     reporting any mismatches as warnings.

     Core Implementation Strategy

     Comparison Philosophy: Lenient Approach

     Static analysis intentionally over-approximates (conservative), while 
     dynamic tracking is precise (only what executed). Therefore:

     Expected behavior (NOT warnings):
     - Static > Dynamic when static has analysis warnings (dynamic code, 
     reflection, etc.)
     - Static > Dynamic due to method dispatch, conditional writes, or 
     transitive closure

     Actual issues (WARN):
     - Critical: Dynamic > Static for reads = soundness violation in static 
     analysis
     - Warning: Dynamic > Static for writes = possible dynamic code 
     execution

     Severity Levels

     - critical - Soundness violation (dynamic found deps static missed)
     - warning - Static missed writes
     - info - Static over-approximated (expected)
     - pass - Perfect match

⏺ Explanation of the Warnings

  The warnings are coming from the conservative nature of static dependency 
  analysis. Here's what's happening:

  The Code

  # Cell usak
  _start = time.time()
  dists = pairwise_distances(pts)  # ← pts passed as argument
  _end = time.time()
  _diff = _end - _start
  print("Time = ", _diff, "sec")   # ← _diff passed as argument

  Why the Warnings Appear

  The static analysis treats all variables passed as function arguments as 
  potentially being called. This is visible at dependencies.py:805-814:

  # Visit arguments - functions/methods passed as arguments are considered 
  called
  for arg in node.args:
      self.visit(arg)
      # Extract callable references (functions or methods)
      func_ref, method_ref = self._extract_callable_reference(arg)
      if func_ref:
          self.functions_called.add(func_ref)  # ← pts and _diff added here

  Reasoning: The static analyzer can't know whether the function will call
  the arguments as callbacks. For example:
  # This is valid Python
  def apply_function(f, x):
      return f(x)  # f is called!

  apply_function(my_func, data)  # my_func is a callback

  So when it sees pairwise_distances(pts) or print("Time = ", _diff, "sec"),
   it conservatively assumes pts and _diff might be functions that get
  called internally.

  Then, during post-processing, it notices:
  - pts is defined as a variable (numpy array), not a function
  - _diff is defined as a variable (float), not a function

  So it warns: "You're 'calling' something that's not a function definition
  - this might be an aliased function (like callback = some_func)."

  Is This a Problem?

  No! This is expected conservative behavior. The warnings indicate:
  1. ✅ Static analysis is being safely conservative (over-approximating)
  2. ✅ Dynamic tracking shows they weren't actually called
  3. ✅ The comparison correctly identifies this as "expected
  over-approximation"

  The verdict was PASS because this mismatch is expected when static
  analysis encounters potential callbacks. This is exactly what your
  test_dependencies command is designed to detect and report appropriately!
