  Key Points

  Problem: When cells execute via JupyterLab's native execution (not through
   execute_all command), the ferret metadata emitted by the kernel is not
  being captured into cell metadata.

  Solution: Add a post-execution hook to ExecutionHookManager that:
  1. Listens for NotebookActions.executed signal
  2. Scans cell outputs for display_data outputs containing profile or
  tracking metadata
  3. Extracts and merges this metadata into cell.model.metadata.ferret

  Data Flow:
  - Kernel emits metadata via display_icon_and_text() → IPython display()
  - Output appears with output.metadata = {profile: {...}, tracking: {...}}
  - Frontend hook extracts this after execution and sets on cell model

  The implementation mirrors the server-side extract_and_set_metadata()
  function from metadata_extractor.py.

     Plan: Extract Ferret Metadata from Kernel Outputs in Frontend

     Overview

     Modify executionhook.ts to look for ferret metadata in cell outputs 
     after execution completes. This mirrors what the server-side
     execute_all command does with extract_and_set_metadata().

     Current Architecture

     Kernel Side (How Metadata is Emitted)

     1. FerretKernel._display_execution_result() (ferret_kernel.py:480-496) 
     creates metadata:
     metadata = {
         'profile': {
             'duration': duration,
             'profile': profile_contents,
             'env': pre_types,
             'env_after': post_types,
         }
     }
     if tracking is not None:
         metadata['tracking'] = tracking
     2. Calls display_icon_and_text() which uses IPython.display.display() 
     with metadata=metadata
     3. This creates a display_data output with the metadata attached to 
     output.metadata

     Server Side (How Metadata is Extracted)

     execute_all.py:121 calls extract_and_set_metadata(cell, 
     result["outputs"]):
     - Loops through outputs looking for output['metadata']
     - Checks for registered keys (profile, tracking)
     - Validates and sets on cell["metadata"]["ferret"]

     Frontend Side (Current State)

     executionhook.ts:
     - Wraps notebook execution commands (notebook:run-cell, etc.)
     - Only intercepts before execution to auto-generate code for string 
     constant cells
     - Does not process execution results or extract metadata

     Implementation Plan

     Step 1: Add Post-Execution Hook

     Extend ExecutionHookManager to listen for cell execution completion 
     using JupyterLab's NotebookActions.executed signal.

     import { NotebookActions } from '@jupyterlab/notebook';

     // In setupHooks():
     NotebookActions.executed.connect(this.onCellExecuted, this);

     Step 2: Create Metadata Extraction Function

     Create a function that mirrors extract_and_set_metadata():

     /**
      * Extract ferret metadata from cell outputs
      */
     private extractFerretMetadata(outputs: IOutput[]): IFerretMetadata | 
     null {
       for (const output of outputs) {
         if (output.output_type !== 'display_data') continue;

         const metadata = (output as any).metadata;
         if (!metadata) continue;

         const ferretMeta: Partial<IFerretMetadata> = {};

         // Extract profile metadata
         if (metadata.profile) {
           ferretMeta.profile = metadata.profile as IFerretProfileData;
         }

         // Extract tracking/dynamic_dependencies metadata
         if (metadata.tracking) {
           ferretMeta.dynamic_dependencies = {
             reads_before_writes: metadata.tracking.reads_before_writes || 
     [],
             writes: metadata.tracking.writes || []
           };
         }

         if (Object.keys(ferretMeta).length > 0) {
           return ferretMeta as IFerretMetadata;
         }
       }
       return null;
     }

     Step 3: Implement Post-Execution Handler

     /**
      * Handle cell execution completion
      */
     private onCellExecuted(
       _sender: any,
       args: { notebook: Notebook; cell: Cell }
     ): void {
       const { cell } = args;
       if (cell.model.type !== 'code') return;

       const outputs = (cell.model as ICodeCellModel).outputs;
       const outputList: IOutput[] = [];
       for (let i = 0; i < outputs.length; i++) {
         outputList.push(outputs.get(i));
       }

       const ferretMeta = this.extractFerretMetadata(outputList);
       if (!ferretMeta) return;

       // Merge with existing ferret metadata
       const existingMeta = cell.model.getMetadata('ferret') as 
     IFerretMetadata || {};
       const mergedMeta = { ...existingMeta, ...ferretMeta };

       cell.model.setMetadata('ferret', mergedMeta);

       console.log(`ExecutionHook: Extracted ferret metadata for cell 
     ${cell.model.id}`);
     }

     Step 4: Add Required Imports

     import { ICodeCellModel, IOutput } from '@jupyterlab/cells';
     import { Notebook, NotebookActions } from '@jupyterlab/notebook';
     import { IFerretMetadata, IFerretProfileData, IDynamicDependencies } 
     from './types';

     Step 5: Cleanup on Dispose

     public dispose(): void {
       NotebookActions.executed.disconnect(this.onCellExecuted, this);
     }

⏺ Error Analysis

  The error occurs because of a dual-storage synchronization bug in
  TrackingDict:

  The Problem

  TrackingDict.__init__(underlying):
      super().__init__(underlying)  # COPIES underlying into parent dict
      self._underlying = underlying  # REFERENCE to original dict

  Critical issue: When dict.copy() is called (by IPython's var_expand):
  1. dict.copy() iterates over the parent dict's keys (from
  super().__init__)
  2. For each key, Python calls __getitem__
  3. __getitem__ looks in _underlying (the original dict reference)
  4. If keys got added to the parent dict but not _underlying, you get
  KeyError

  How _i7 Got Out of Sync

  When %enable_global_tracking runs:
  1. IPython updates _i7 (input history) using internal methods
  2. Some IPython code might call dict.update() or dict.__setitem__()
  directly on the parent class
  3. TrackingDict doesn't override update(), setdefault(), etc.
  4. Result: parent dict has _i7, but _underlying doesn't

  ---
  Options

  Option A: Single Storage (Simplest Fix)

  Don't keep _underlying reference. Just use the parent dict directly.

  class TrackingDict(dict):
      def __init__(self, initial_ns):
          super().__init__(initial_ns)
          self._initial_keys = set(self.keys())
          self.reset_tracking()

      def __getitem__(self, key):
          val = super().__getitem__(key)  # Use parent dict
          if not key.startswith('_') and key not in self._writes:
              self._reads_before_writes.add(key)
          return val

      def __setitem__(self, key, value):
          if key not in self._writes:
              self._writes.add(key)
              if key not in self._initial_keys:
                  self._new_vars.add(key)
          super().__setitem__(key, value)

  Pros: Simple, no sync issues, all dict methods work
  Cons: Original user_ns object is replaced (but ipython.user_ns is
  canonical)

  ---
  Option B: Use __class__ Injection (Preserve Original Dict)

  Change the class of the existing dict instead of wrapping:

  def enable_tracking(ns):
      # Inject tracking methods into the existing dict
      ns.__class__ = TrackingDict  # Won't work - can't change built-in type

  This doesn't work for built-in dict, but could work with a MutableMapping
  wrapper.

  ---
  Option C: Pre/Post Snapshot Approach

  Don't modify user_ns at all. Compare before/after snapshots:

  def pre_run_cell(info):
      ip = get_ipython()
      ip._ferret_snapshot = set(ip.user_ns.keys())

  def post_run_cell(result):
      ip = get_ipython()
      old_keys = ip._ferret_snapshot
      new_keys = set(ip.user_ns.keys())
      added = new_keys - old_keys
      removed = old_keys - new_keys

  Pros: Zero risk of breaking IPython, simple
  Cons: Can't track reads-before-writes (only new/deleted vars)

  ---
  Option D: AST-Based Only

  Use static AST analysis (which you already have) instead of runtime
  tracking.

  Pros: No runtime overhead, no IPython internals risk
  Cons: Can't handle eval(), exec(), globals()[name]

  ---
  Option E: Override ALL Dict Methods

  Make TrackingDict properly proxy everything:

  class TrackingDict(dict):
      def update(self, *args, **kwargs):
          # Track all updates
          super().update(*args, **kwargs)

      def setdefault(self, key, default=None):
          # Track setdefault
          ...

      def __ior__(self, other):  # |= operator
          ...

  Pros: Keeps original reference working
  Cons: Complex, easy to miss methods, fragile

  ---
  Recommendation

  Option A (Single Storage) is the cleanest fix. The reason for keeping
  _underlying was to allow external code holding the old user_ns reference
  to still work. But in practice:

  1. IPython's canonical reference is ipython.user_ns
  2. The ferret kernel controls when tracking is enabled
  3. Synchronization bugs are worse than replacing the dict
