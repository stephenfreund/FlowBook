  Problems Identified

  Issue 1: Nested DataFrames/Series
  When a DataFrame or Series is stored inside an object's instance variable,
   it bypasses the special handling in _deep_copy_user_ns(). The standard
  copy.deepcopy() is used instead, which:
  - Doesn't properly deep copy object-dtype columns with mutable objects
  - Doesn't apply the immutable column optimization
  - Can fail with pandas 2.3+ immutable column errors

  Issue 2: Functions Inside Objects
  Functions stored as instance variables don't get the _deep_copy_function()
   treatment, so:
  - Closure contents aren't deep copied
  - Mutable default arguments aren't properly isolated

  Root Cause: The special handling in _deep_copy_user_ns() only applies to
  top-level variables. When it encounters nested structures, it falls back
  to copy.deepcopy(v, memo=memo), which doesn't apply custom logic.

  Solution Options Assessed

  1. Custom Recursive Deepcopy (RECOMMENDED) - Create a custom deepcopy that
   intercepts DataFrames/Series/Functions at any nesting level
  2. Monkey-Patch pandas - Add custom __deepcopy__ to pandas classes (risky,
   global side effects)
  3. Two-Pass Approach - Standard deepcopy then fix problematic objects
  (inefficient, complex)
  4. Minimal Fix - Handle one level of nesting only (low risk fallback)

  Recommended Approach

  Primary: Full Custom Recursive Deepcopy

  Create a _custom_deepcopy() function that:
  - Checks memo first (circular references)
  - Intercepts DataFrames → _deep_copy_dataframe()
  - Intercepts Series → _deep_copy_series()
  - Intercepts Functions → _deep_copy_function()
  - Recursively handles containers (list, dict, tuple, set)
  - Recursively handles objects with __dict__
  - Falls back to standard copy.deepcopy() for everything else

  This ensures proper handling at ANY nesting depth while maintaining memo
  consistency for circular references.

  Benefits:
  - ✅ Handles all nesting levels correctly
  - ✅ No global side effects
  - ✅ Maintains all current optimizations (CoW, immutable skipping)
  - ✅ Extensible for future special cases

  Risks:
  - Complexity in implementation
  - Need comprehensive testing
  - Slight performance overhead from type checking

  Fallback: Minimal Fix - If the full solution is too complex, implement
  one-level nesting support by adding _deep_copy_object_dict() that checks
  instance attributes for DataFrames/Series/Functions.

  How It Works

  Before (slow):
  - EVERY object with __dict__ → manual attribute-by-attribute copying in
  Python

  After (optimized):
  - Objects with only primitive attributes → fast C-level copy.deepcopy()
  - Objects containing custom objects/DataFrames/Series/Functions → manual
  copying (necessary)

  The Fast Path Check

  The _contains_special_types() function does a quick scan of the object's
  __dict__:

  1. ✅ Fast path (delegates to standard deepcopy):
    - Objects with only primitives: int, str, float, bool, None
    - Objects with lists/dicts of primitives
    - NumPy arrays (handled efficiently by standard deepcopy)
  2. ⚠️ Slow path (manual copying needed):
    - Objects containing DataFrames/Series/Functions
    - Objects containing other custom objects (might contain special types)

  Why We Can't Fully Avoid Manual Copying

  We cannot delegate to standard copy.deepcopy() for objects with nested
  custom objects because:
  - Standard deepcopy won't use our _custom_deepcopy() logic recursively
  - If it encounters a nested DataFrame, it won't apply our special
  object-column handling
  - We'd lose the whole benefit of the custom deepcopy system

  Performance Characteristics

  Common case (leaf objects with primitives): Near-native speed

  # ✅ Uses fast standard deepcopy

  Nested objects: Still manual copying, but unavoidable
  class Container:
      config = Config()  # Custom object
      df = pd.DataFrame()  # Special type  
  # ⚠️ Uses manual copying (necessary for correctness)

  Net Result

  - Objects without nested special types: ~same performance as before
  - Objects with nested DataFrames/Series/Functions: Works correctly

  - No regression for the common case, fixes the correctness issue for
