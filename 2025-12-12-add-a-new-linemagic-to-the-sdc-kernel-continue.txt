⏺ Performance Investigation Summary

  I identified several performance bottlenecks introduced by the CompoundDiff changes:

  1. CRITICAL: pprint.pformat() in _format_diff_for_display (sdc_enforcer.py:604)

  This is the primary bottleneck. When truncation is detected in a diff:
  - _check_for_truncation calls _format_diff_for_display
  - Which calls pprint.pformat() on the converted diff structure
  - Result: 7.5 seconds for 1000 calls on a typical truncated DataFrame diff

  | Scenario        | Time (10k calls)        |
  |-----------------|-------------------------|
  | No truncation   | 0.002s                  |
  | With truncation | 25.4s (~12,700x slower) |

  The pprint call alone accounts for 97% of the time when truncation occurs.

  2. MODERATE: Pydantic Model Creation Overhead

  | Operation         | Time (10k) | vs Plain Dict |
  |-------------------|------------|---------------|
  | CompoundDiff()    | 0.007s     | ~10x slower   |
  | ValueComparison() | 0.010s     | ~11x slower   |

  This is a baseline overhead of ~0.015s per 5000 model creations - noticeable but not severe for typical diffs.

  3. MINOR: model_post_init defensive checks in ValueComparison

  The float tolerance validation in model_post_init (types.py:95-153) adds only ~10% overhead when comparing floats. Not a significant bottleneck.

  ---
  Recommended Improvements

  High Priority:
  1. Lazy formatting in _check_for_truncation - Don't format the diff until it's actually needed for display. Return just the truncated var names, compute the formatted string only when displayed.
  2. Replace pprint.pformat() with a simpler formatter - Use a fast custom formatter or skip the recursive pformat entirely for SDC violation messages.

  Medium Priority:
  3. Use model_construct() for internal model creation - Pydantic's model_construct() skips validation and is ~3-5x faster:
  # Instead of:
  CompoundDiff(source_type="array", children={}, truncated=False)
  # Use:
  CompoundDiff.model_construct(source_type="array", children={}, truncated=False)

  4. Consider dataclasses instead of Pydantic for hot paths - If validation isn't needed internally, plain dataclasses are much faster.

  Low Priority:
  5. Cache isinstance() results - The code has many isinstance(node, CompoundDiff) checks, though benchmarks show this isn't a significant bottleneck (~0.004s/100k).

  6. Consider removing model_post_init defensive check - It's only useful during development/debugging. Could be disabled via a flag in production.
